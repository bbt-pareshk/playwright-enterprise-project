## 🏗️ 1. Architecture Overview (The "Why" and "Where")

This framework utilizes a strictly layered **Enterprise Page Object Model (POM)**. To ensure high speed and zero flakiness, responsibilities belong to specific directories. **No business logic exists in `specs/`.**

*   **`config/`**: Centralized environment configuration (`env.ts`, `urls.ts`).
*   **`lib/pages/`**: Atomic Page Objects (Locators + strict visual/state assertions). Pages do *not* call other pages.
*   **`lib/flows/`**: Orchestration for multi-stage journeys that stay within a specific domain (e.g., `GroupFlow`, `SessionFlow`).
*   **`lib/helpers/`**: High-level cross-domain services (e.g., `LeaderHelper`, `AuthHelper`, `PaymentHelper`).
*   **`lib/utils/`**: Technical utilities independent of the application (`Logger`, `DataGenerator`, `VerificationService`).
*   **`lib/data/constants/`**: **Critical Folder**. *Zero Magic Strings.* All labels, errors, and hardcoded routes live here (`ui-constants.ts`, `app-constants.ts`, `messages.ts`).
*   **`specs/`**: Test Specifications. Highly readable files focusing on assertions and calling `Helper` or `Flow` methods.
*   **`storage/`**: State files (cookies, tokens) generated by `setup/` scripts for reusing authenticated sessions across tests.
*   **`DOM/` & `DOM Files/`**: Captured historical artifacts of the UI. *Do not delete during cleanups.*

## 🛡️ 2. Enterprise Defense Mechanisms (CRITICAL GUIDELINES)

This framework achieves 100% stability by employing active defense strategies against common E2E flakiness. **Any Developer or AI modifying the code MUST adhere to these rules.**

### 🚨 A. The "Enterprise Kill-Switch" (Third-Party Blocking)
Support widgets like Gleap, Chameleon, and Intercom inject Shadow DOMs that overlay application buttons, causing tests to fail with `TimeoutError: ... waiting for element to be visible`.
*   **How it works:** `lib/fixtures/base.fixture.ts` actively aborts network requests to these domains and injects CSS to `display: none` any latent containers.
*   **The Rule:** Standard tests using the `{ page, context }` fixtures get this automatically. However, **if a test creates a manual context** via `browser.newContext()` to share state across serial blocks, you **MUST** apply the kill-switch manually:
    ```typescript
    import { applyEnterpriseContextSettings } from '../../../lib/fixtures/base.fixture';
    
    test.beforeAll(async ({ browser }, testInfo) => {
        context = await browser.newContext();
        await applyEnterpriseContextSettings(context, testInfo); // MANDATORY
        page = await context.newPage();
    });
    ```

### 🧠 B. Adaptive Synchronization ("Empty State" Handling)
Applications render differently for new accounts versus established accounts (e.g., "My Groups" tabs missing vs showing "You haven't joined any groups yet"). Hardcoded waits for established elements will fail on fresh users.
*   **The Rule:** When validating page load parity, use Playwright's `locator.or()` to acknowledge either condition as "Loaded".
    ```typescript
    const tabs = this.page.getByRole('tablist');
    const emptyState = this.page.getByText(UI_CONSTANTS.GROUPS.EMPTY_STATE);
    
    // Waives the timeout risk by accepting the Empty State as valid parity
    await tabs.or(emptyState).waitFor({ state: 'visible', timeout: 30_000 });
    ```

### 🔒 C. Session Leak Prevention (Smart Auto-Login Bypassing)
Modern SPAs automatically detect left-over cookies and redirect aggressive URL hits (e.g. going to `/login` automatically forwards to `/dashboard`).
*   **The Rule:** Any test focusing on anonymous user capabilities (Registration Matrix, Forgot Password, Logged Out visibility) MUST begin with a proactive purge.
    ```typescript
    test.beforeEach(async ({ page }) => {
        await AuthHelper.forceLogout(page); // Clears all storage and cookies
    });
    ```

### 🌊 D. "Deep Check" UI Progression Integration
For tests that rely on pre-requisite sequences (like Onboarding), we don't blindly click buttons. 
*   **The Rule:** Check if the destination is already reached before executing the UI steps. (e.g., in `OnboardingPage.completeLeaderOnboardingViaSkip()`, if the URL is already `/hosting-plan`, immediately return).

## ⚡ 3. Test Design & Execution Optimization

### Execution Flow Matrix (`playwright.config.ts`)
*   **`setup-auth`**: Provisions fundamental baseline accounts.
*   **`setup-leader-functional`**: Provisions advanced state (a leader with active slots/payments) stored into `storage/auth/leader_functional.json`.
*   **`group-v2`**: (Serial) Heavy lifecycle flows utilizing reused state.
*   **`default`**: Highly parallelized atomic regression specs (Forms, Auth, Matrix logic).

### Test Logic & Structure Constraints
1.  **Never Use `waitForTimeout`**: Rely strictly on visual signals, network idle states, or `Promise.race()` handlers for dynamic spinners/toast messages.
2.  **No Direct Value Assertions in Specs**:
    *   ❌ Bad: `expect(locator).toHaveText("Welcome, John")`
    *   ✅ Good: `expect(locator).toHaveText(new RegExp(UI_CONSTANTS.AUTH.LOGIN.HEADING))`
3.  **Atomic Flow Isolation**: In `.serial` blocks, ensure tests do not strictly rely on visual dom state left by the *previous* test unless absolutely necessary, or else handle early-return logic graciously.

## 🧹 4. Maintenance & Cleanup Policy
To keep the execution pipeline and local DevX fast and light:
*   Artifacts like `/test-results/`, `/allure-results/`, `/playwright-report/`, and `tmp/` are considered transient. They are aggressively removed automatically across runs.
*   Do **NOT** remove the raw `DOM/` folders, as they are used to audit locator regressions when the upstream UI changes.
*   If replacing an old multi-file feature with a new Flow architecture (e.g., migrating 5 scattered group creation tests to `group-lifecycle-v2.spec.ts`), **DELETE** the old specs outright. Dead code rots.
